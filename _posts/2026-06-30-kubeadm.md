## Summary

kubeadm is the cluster-lifecycle tool: bootstrap, join, upgrade, certs, tokens, reset. On the CKA it shows up most in Cluster Architecture, Installation & Configuration (25%) — the upgrade workflow and `--print-join-command` are near-certain tasks.

## Key concepts

Almost every subcommand supports `--help` and phase-level breakdowns (`kubeadm <cmd> phase --help`). Phases let you run one step in isolation, which is useful for troubleshooting. Most subcommands need `sudo`.

| Command           | Purpose                                                |
| ----------------- | ------------------------------------------------------ |
| `kubeadm init`    | Bootstrap a control-plane node                         |
| `kubeadm join`    | Join a node (worker or control-plane) to the cluster   |
| `kubeadm upgrade` | Upgrade the cluster components                         |
| `kubeadm token`   | Manage bootstrap tokens (and regenerate join commands) |
| `kubeadm certs`   | Inspect/renew/manage PKI certificates                  |
| `kubeadm config`  | View/migrate config, list/pull required images         |
| `kubeadm reset`   | Tear down what init/join changed on a node             |

### kubeadm init

Bootstraps the first control-plane node.

```bash
sudo kubeadm init \
  --pod-network-cidr=10.244.0.0/16 \
  --apiserver-advertise-address=10.0.0.10 \
  --control-plane-endpoint=10.0.0.10:6443 \
  --kubernetes-version=v1.35.0 \
  --upload-certs \
  --cri-socket=unix:///var/run/containerd/containerd.sock
```

Key flags:
- `--pod-network-cidr` — must match what your CNI expects.
- `--control-plane-endpoint` — set this if you'll ever have >1 control plane (can't be added cleanly later); point at a load balancer or stable DNS.
- `--upload-certs` — uploads control-plane certs as a Secret so additional CP nodes can pull them; pairs with `--certificate-key`. Keys expire after ~2h.
- `--config <file>` — drive everything from a `ClusterConfiguration`/`InitConfiguration` YAML instead of flags.

Post-init (set up kubectl):

```bash
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

Run individual init phases when troubleshooting:

```bash
sudo kubeadm init phase --help
sudo kubeadm init phase certs all
sudo kubeadm init phase upload-certs --upload-certs   # re-upload CP certs later
```

### kubeadm join

Joins a node. The token + CA hash come from `init` output or from regenerating them (see `token`).

```bash
# Worker
sudo kubeadm join 10.0.0.10:6443 \
  --token abcdef.0123456789abcdef \
  --discovery-token-ca-cert-hash sha256:<hash>

# Additional control-plane node (HA)
sudo kubeadm join 10.0.0.10:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane \
  --certificate-key <cert-key>
```

Compute the CA cert hash manually if you lost the init output:

```bash
openssl x509 -pubkey -in /etc/kubernetes/pki/ca.crt \
  | openssl rsa -pubin -outform der 2>/dev/null \
  | openssl dgst -sha256 -hex | sed 's/^.* //'
```

### kubeadm token

Bootstrap tokens authorize new nodes to join. Default TTL is 24h, so on the exam you often need to regenerate before joining a node.

```bash
kubeadm token list
sudo kubeadm token create                  # create and print a new token
sudo kubeadm token create --ttl 2h        # custom lifetime (0 = never expires)
sudo kubeadm token delete <token>
```

The one to memorize — regenerate a complete, ready-to-paste join command:

```bash
sudo kubeadm token create --print-join-command
```

This prints the full `kubeadm join ... --token ... --discovery-token-ca-cert-hash sha256:...` line. For a CP join, append `--control-plane --certificate-key <key>`.

### kubeadm upgrade

Upgrade order: **first control-plane node → other CP nodes → workers**. Always upgrade the `kubeadm` binary first on each node, one minor version at a time.

```bash
sudo kubeadm upgrade plan                 # read-only: shows current vs available
sudo kubeadm upgrade apply v1.35.1        # first CP node only
sudo kubeadm upgrade node                 # every other node (additional CP + workers)
sudo kubeadm upgrade diff v1.35.1         # preview manifest changes
```

`upgrade node` phase breakdown:

```bash
sudo kubeadm upgrade node phase control-plane    # CP components on extra CP nodes
sudo kubeadm upgrade node phase kubelet-config   # syncs kubelet-config ConfigMap → local file
```

Note: `kubelet-config` phase writes the cluster-wide ConfigMap down to `/var/lib/kubelet/config.yaml`. Local hand-edits to that file are overwritten — the ConfigMap is the source of truth.

### kubeadm certs

Manages the control-plane PKI (apiserver, etcd, front-proxy, etc.). Does **not** cover kubelet client/serving certs — those rotate via the CSR mechanism separately.

```bash
sudo kubeadm certs check-expiration       # table of all managed certs + expiry
sudo kubeadm certs renew all              # renew every managed cert
sudo kubeadm certs renew apiserver        # renew a specific cert
sudo kubeadm certs certificate-key        # generate a new --certificate-key for HA joins
```

After `certs renew`, restart the static-pod control plane so processes pick up the new certs (they read certs at start). Move manifests out and back:

```bash
cd /etc/kubernetes/manifests
sudo mkdir -p /tmp/m && sudo mv *.yaml /tmp/m/ && sleep 20 && sudo mv /tmp/m/*.yaml .
```

Renewals also happen automatically on `kubeadm upgrade apply`.

### kubeadm config

```bash
kubeadm config print init-defaults        # template InitConfiguration/ClusterConfiguration
kubeadm config print join-defaults
sudo kubeadm config images list           # images this version needs
sudo kubeadm config images pull           # pre-pull (useful before init in air-gapped envs)
```

### kubeadm reset

Undo what init/join did on a node — use before re-joining a broken node.

```bash
sudo kubeadm reset
sudo rm -rf /etc/cni/net.d
sudo rm -rf $HOME/.kube/config
sudo iptables -F && sudo iptables -t nat -F && sudo iptables -t mangle -F && sudo iptables -X
```

### Important paths

| Path                                           | What it is                                                            |
| ---------------------------------------------- | --------------------------------------------------------------------- |
| `/etc/kubernetes/manifests/`                   | Static pod manifests (apiserver, controller-manager, scheduler, etcd) |
| `/etc/kubernetes/pki/`                         | Control-plane certificates and keys                                   |
| `/etc/kubernetes/pki/etcd/`                    | etcd certificates                                                     |
| `/etc/kubernetes/admin.conf`                   | Cluster-admin kubeconfig                                              |
| `/etc/kubernetes/kubelet.conf`                 | kubelet's kubeconfig                                                  |
| `/var/lib/kubelet/config.yaml`                 | kubelet runtime config (synced from `kubelet-config` ConfigMap)       |
| `/var/lib/etcd/`                               | etcd data directory (default)                                         |

## Code

Full upgrade sequence — memorize this rhythm.

**Control-plane (first node):**

```bash
# 1. upgrade kubeadm binary
sudo apt-mark unhold kubeadm
sudo apt-get update && sudo apt-get install -y kubeadm=1.35.1-*
sudo apt-mark hold kubeadm

# 2. plan + apply
sudo kubeadm upgrade plan
sudo kubeadm upgrade apply v1.35.1

# 3. drain
kubectl drain <cp-node> --ignore-daemonsets

# 4. upgrade kubelet + kubectl, restart kubelet
sudo apt-mark unhold kubelet kubectl
sudo apt-get update && sudo apt-get install -y kubelet=1.35.1-* kubectl=1.35.1-*
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet

# 5. uncordon
kubectl uncordon <cp-node>
```

**Worker node (repeat per worker):**

```bash
# 1. upgrade kubeadm binary
sudo apt-mark unhold kubeadm
sudo apt-get update && sudo apt-get install -y kubeadm=1.35.1-*
sudo apt-mark hold kubeadm

# 2. upgrade local node config
sudo kubeadm upgrade node

# 3. drain (from a node where kubectl is configured)
kubectl drain <worker> --ignore-daemonsets

# 4. upgrade kubelet + kubectl, restart (ON the worker)
sudo apt-mark unhold kubelet kubectl
sudo apt-get update && sudo apt-get install -y kubelet=1.35.1-* kubectl=1.35.1-*
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet

# 5. uncordon
kubectl uncordon <worker>
```

**etcd backup & restore** (etcdctl, not kubeadm — but adjacent and exam-tested):

```bash
# backup
sudo ETCDCTL_API=3 etcdctl snapshot save /opt/snapshot.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# verify
sudo ETCDCTL_API=3 etcdctl snapshot status /opt/snapshot.db --write-out=table

# restore to a new data dir
sudo ETCDCTL_API=3 etcdctl snapshot restore /opt/snapshot.db \
  --data-dir=/var/lib/etcd-restore

# point etcd static pod at the new data dir:
#   edit /etc/kubernetes/manifests/etcd.yaml
#   update --data-dir and the volume hostPath to /var/lib/etcd-restore
#   kubelet recreates the etcd pod; control plane comes back on restored data
```

## Gotchas

- Upgrade order matters: always upgrade the `kubeadm` binary on a node before running `upgrade apply`/`upgrade node` on it — not after.
- `upgrade apply` is first-CP-node-only; every other node uses `upgrade node`. Running `apply` on a second CP node will fail or do the wrong thing.
- Drain before upgrading the kubelet, uncordon after — skipping this leaves the node unschedulable.
- After `certs renew`, bounce the static pods by moving manifests out and back — a kubelet restart alone isn't enough because the processes are already running with old file handles.
- `check-expiration` only shows the kubeadm-managed PKI, not kubelet serving certs.
- Bootstrap tokens expire in 24h by default. If a join fails with an auth error, regenerate with `--print-join-command`.
- `--control-plane-endpoint` can't be cleanly added after init — set it at bootstrap time if there's any chance of adding more control-plane nodes later.
- `kubelet-config` phase overwrites `/var/lib/kubelet/config.yaml` from the ConfigMap — local edits there don't survive an upgrade.
